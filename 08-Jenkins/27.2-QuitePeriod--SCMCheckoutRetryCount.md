# Quiet Period

## What is Quiet Period?

The **Quiet Period** is the amount of time Jenkins waits **before starting a build** after receiving a build request.

Instead of executing the build immediately, Jenkins delays it for a specified number of seconds.

This delay allows Jenkins to collect multiple build requests and avoid triggering unnecessary builds.

---

# Why Do We Need a Quiet Period?

Developers often push multiple commits within a few seconds or minutes.

Without a quiet period, Jenkins starts a new build for every commit.

# When Should You Use It?

A Quiet Period is useful when:

- Developers push multiple commits quickly.
- Large monorepositories generate frequent builds.
- Builds are resource intensive.
- You want to avoid duplicate pipeline executions.

---

# When Should You Avoid It?

A Quiet Period may not be suitable for:

- Emergency hotfix deployments.
- Production release pipelines.
- Manual approval pipelines.
- Time-sensitive automation.

These builds often need to start immediately.

# Best Practices

- Keep the Quiet Period short for development pipelines.
- Use longer delays only for repositories with frequent commits.
- Set the Quiet Period to **0** for critical production deployments when immediate execution is required.
- Monitor build frequency and adjust the value based on actual usage.
- Combine the Quiet Period with webhooks for efficient pipeline triggering.

# SCM Checkout Retry Count

## What is SCM Checkout Retry Count?

The **SCM Checkout Retry Count** determines how many times Jenkins should retry checking out source code from the Source Code Management (SCM) system before marking the build as failed.

Sometimes Git checkouts fail due to temporary issues like:

- Network interruptions
- Git server overload
- Temporary authentication failures
- DNS resolution problems
- Short-lived connectivity issues

Instead of immediately failing the build, Jenkins retries the checkout automatically.

---

# Why Do We Need SCM Checkout Retry Count?

In production environments, source code is usually stored in remote repositories like:

- GitHub
- GitLab
- Bitbucket
- Azure Repos

Network communication isn't always perfect.

Without retries, even a small network issue can fail an entire CI/CD pipeline.
