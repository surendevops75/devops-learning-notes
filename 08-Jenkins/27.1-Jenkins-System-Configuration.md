# Global Environment Variables

## What are Global Environment Variables?

Global Environment Variables are **key-value pairs** configured once in Jenkins and automatically made available to all jobs, pipelines, and build agents.

Instead of defining the same environment variable in every Jenkinsfile, administrators can configure it globally so every project can reuse it.

These variables are available throughout the Jenkins instance unless overridden at the job or pipeline level.

---

# Why Do We Need Global Environment Variables?

In enterprise environments, multiple pipelines often require the same configuration values.

For example:

- AWS Region
- Docker Registry URL
- Kubernetes Namespace
- SonarQube Server URL
- Terraform Environment
- Company Name

Rather than duplicating these values across hundreds of pipelines, Jenkins allows administrators to configure them once.


# Variable Precedence

```text
Pipeline Stage Variable

↓

Pipeline Variable

↓

Job Variable

↓

Folder Variable

↓

```

If multiple variables have the same name, Jenkins uses the highest-priority definition.

- AWS Secret Keys

# Key Points

- Global Environment Variables are shared across the Jenkins instance.
- They reduce duplication by allowing administrators to define common values once.
- Pipelines can directly reference these variables using standard environment syntax.

# Number of Executors

## What are Executors?

An **Executor** is a worker thread inside Jenkins that is responsible for running a build or pipeline.

Think of an executor as a **slot** where Jenkins can execute one build at a time.

If a node has:

- 1 Executor → Runs 1 build simultaneously
- 2 Executors → Runs 2 builds simultaneously
- 4 Executors → Runs 4 builds simultaneously

Each executor can execute only one job at a time.

---

# Why Do We Need Executors?

In enterprise environments, multiple developers trigger builds simultaneously.

Without multiple executors,

```text
Developer A

↓

Build Running

↓

Developer B

↓

Waiting

↓

Developer C

↓

Waiting
```

This increases the build queue and delays deployments.

Executors allow Jenkins to process multiple jobs concurrently.

---

# Build Queue Without Enough Executors

```text
                Jenkins

                   │

                   ▼

             Build Queue

      ┌──────────┬──────────┬──────────┐

      │ Build 1  │ Build 2  │ Build 3  │

      └──────────┴──────────┴──────────┘

                   │

                   ▼

             1 Executor Only

                   │

                   ▼

          Build 1 Executes

          Build 2 Waits

          Build 3 Waits
```

Problems

- Long waiting time
- Slow CI/CD
- Idle developers
- Delayed releases

---

# Multiple Executors

```text
                 Jenkins

                    │

                    ▼

              Build Queue

        ┌────────┬────────┬────────┐

        │Build 1 │Build 2 │Build 3 │

        └────────┴────────┴────────┘

                    │

      ┌─────────────┼─────────────┐

      ▼             ▼             ▼

 Executor 1    Executor 2    Executor 3

      │             │             │

      ▼             ▼             ▼

 Build 1       Build 2       Build 3
```

Benefits

- Faster builds
- Better utilization
- Reduced queue
- Improved developer productivity

# Controller vs Agent Executors

```text
              Jenkins Controller

             Executors = 0

                   │

        Schedules Builds Only

                   │

                   ▼

          Build Agents

      ┌────────────┬────────────┐

      ▼            ▼            ▼

 Agent A      Agent B      Agent C

Executors=2  Executors=4  Executors=8
```

In production, builds should run on agents rather than the controller.

---

# Why Set Controller Executors to 0?

Production best practice:

```text
Controller

↓

No Build Execution

↓

Only Scheduling

↓

Agents Execute Jobs
```

Advantages

- Better controller stability
- Faster UI response
- Improved scalability
- Easier maintenance

# Best Practices

- Set controller executors to **0** in production.
- Run builds on dedicated agents.
- Size executors based on CPU and memory.
- Continuously monitor build queue length.
- Avoid assigning more executors than the hardware can efficiently support.
- Scale horizontally by adding agents instead of increasing executors excessively.

# Key Points

- An executor is a worker thread that runs one build at a time.
- Multiple executors allow Jenkins to run multiple builds concurrently.
- Insufficient executors create build queues and increase pipeline execution time.
- In production, configure the controller with **0 executors** and run builds on dedicated agents.
- Choose executor counts based on available hardware resources and workload characteristics.
