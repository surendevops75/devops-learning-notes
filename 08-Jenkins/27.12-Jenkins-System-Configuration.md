# Interview Questions

## Question 1

### What is Global Tool Configuration in Jenkins?

### Production-Level Answer

Global Tool Configuration allows administrators to centrally configure tools such as JDK, Maven, Docker, NodeJS, Terraform, and SonarQube scanners. Pipelines can reference these tools by name instead of hardcoding installation paths, making builds consistent across multiple agents and simplifying upgrades.

### Follow-Up Questions

- Where is Global Tool Configuration located?
- What tools can be configured globally?
- Why shouldn't tool paths be hardcoded?

---

## Question 2

### Why should tools be configured globally instead of inside every Jenkinsfile?

### Production-Level Answer

Global configuration ensures all pipelines use the same tool versions, reduces duplication, simplifies maintenance, and allows administrators to upgrade tools without modifying individual Jenkinsfiles.

### Follow-Up Questions

- What happens if different agents use different Maven versions?
- How does this improve maintainability?
- Which tool versions should be standardized?

---

## Question 3

### How do you securely manage credentials in Jenkins?

### Production-Level Answer

Sensitive information such as GitHub Personal Access Tokens, AWS credentials, Docker registry passwords, SonarQube tokens, Slack bot tokens, and SSH keys should be stored in Jenkins Credentials. Pipelines reference these credentials securely using credential IDs instead of exposing secrets in source code.

### Follow-Up Questions

- Which credential types does Jenkins support?
- Why is hardcoding secrets dangerous?
- How do pipelines access stored credentials?

---

## Question 4

### Why is SonarQube configured globally?

### Production-Level Answer

Global SonarQube configuration allows all pipelines to connect to the same SonarQube server using centrally managed authentication tokens. This ensures consistent static code analysis and Quality Gate enforcement across projects.

### Follow-Up Questions

- What is a Quality Gate?
- Where is the SonarQube token stored?
- What happens if the Quality Gate fails?

---

## Question 5

### How is Docker configured in Jenkins?

### Production-Level Answer

Docker is installed on the Jenkins controller or build agents rather than configured through Global Tool Configuration. Jenkins executes Docker CLI commands to build, tag, scan, and push container images. Registry credentials are securely managed using Jenkins Credentials.

### Follow-Up Questions

- Why is Docker usually installed on build agents?
- How does Jenkins authenticate with Docker registries?
- Why should Docker images be scanned before pushing?

---

## Question 6

### Why are dynamic cloud agents preferred over static agents?

### Production-Level Answer

Dynamic agents are provisioned only when builds are required and terminated afterward. This improves scalability, reduces infrastructure costs, and provides clean, isolated build environments. Kubernetes Pods and Amazon EC2 instances are commonly used as dynamic Jenkins agents.

### Follow-Up Questions

- What is the lifecycle of a Kubernetes build agent?
- Why shouldn't the Jenkins controller execute builds?
- Which cloud providers are commonly used?

---

## Question 7

### Why should tool versions be standardized across all Jenkins agents?

### Production-Level Answer

Using different tool versions can produce inconsistent build results, dependency conflicts, or deployment failures. Standardizing versions of JDK, Maven, NodeJS, Terraform, and Docker ensures reproducible builds regardless of the build agent executing the pipeline.

### Follow-Up Questions

- What problems occur when agents use different JDK versions?
- How can version mismatches affect Maven builds?
- How are tool versions upgraded safely?

## Question 9

### How do you troubleshoot a pipeline failure caused by tool configuration?

### Production-Level Answer

First, review the Console Output to identify the failing stage. Verify that the required tool is installed and correctly configured in Global Tool Configuration. Confirm that the Jenkinsfile references the correct tool name and that the build agent has access to the tool.

### Follow-Up Questions

- Which logs do you check first?
- How do you verify a tool installation?
- What commands help validate tool versions?

### Production-Level Answer

GitHub Webhook

↓

Jenkins Trigger

↓

Provision Build Agent

↓

Load Tools

↓

Load Credentials

↓

Git Checkout

↓

Compile

↓

Unit Tests

↓

SonarQube Scan

↓

Quality Gate

↓

↓

↓

↓

Deploy to Amazon EKS

↓

↓

# Key Takeaways

- **System Configuration** is the foundation of every production Jenkins environment.
- Configure **Jenkins URL**, **Executors**, **Shell**, and **Git** before creating pipelines.
- Manage development tools such as **JDK**, **Maven**, **Docker**, **NodeJS**, **Terraform**, and **SonarQube** from **Global Tool Configuration**.
- Store all sensitive information in **Jenkins Credentials** instead of hardcoding secrets in Jenkinsfiles.
- Configure **Email** and **Slack** notifications to keep development and operations teams informed about pipeline events.
- Use **dynamic cloud agents** (Kubernetes Pods or EC2 instances) instead of running builds on the Jenkins controller.
- Standardize tool versions across all build agents to ensure consistent and reproducible builds.
- Integrate **SonarQube**, **Trivy**, and **Checkov** into CI/CD pipelines to improve code quality and security.
- Use **RBAC**, **HTTPS**, **least-privilege IAM permissions**, and **regular backups** to secure Jenkins.
- Monitor Jenkins health, plugin updates, agent status, and build performance to maintain a reliable CI/CD platform.
- Follow a structured troubleshooting process by checking the **Console Output**, **Jenkins Logs**, **Tool Configuration**, **Credentials**, and **Agent Status** before investigating application code.
- A properly configured Jenkins environment provides a **secure**, **scalable**, and **production-ready** CI/CD platform capable of supporting enterprise DevOps workflows.
