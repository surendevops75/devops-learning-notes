# JDK Configuration

## What is JDK Configuration?

**JDK Configuration** allows Jenkins administrators to configure one or more Java Development Kit (JDK) installations that can be used by pipelines and build jobs.

Many applications such as:

- Java
- Spring Boot
- Maven
- Gradle
- SonarQube Scanner

require a JDK to compile and execute Java code.

Instead of specifying the Java installation path in every pipeline, Jenkins provides centralized JDK management.

---

# Why Do We Need JDK Configuration?

A Java application cannot be compiled without a JDK.

Consider a Spring Boot application.

```text
Developer

↓

GitHub

↓

Jenkins

↓

Compile Java

↓

Build JAR

↓

Docker Image

↓

Deploy
```

If no JDK is available,

the build stops immediately.

# Example Jenkinsfile

```groovy
pipeline {

    agent any

    tools {

        jdk 'JDK-21'

    }

    stages {

        stage('Verify Java') {

            steps {

                sh '''
                java -version
                javac -version
                '''
            }

        }

    }

}
```

Jenkins automatically sets

```text
JAVA_HOME
```

and updates the system

```text
PATH
```

for the pipeline.

# Common Mistakes

## Java Not Installed

Pipeline

```text
Java Not Found
```

Verify

```bash

```

---

## Wrong JAVA_HOME

Incorrect

```text
/usr/java
```

Correct

```text
/usr/lib/jvm/java-21-openjdk
```

Verify

```bash
echo $JAVA_HOME
```

---

## Different JDK Versions

```text
Agent A

JDK-17

----------------------

Agent B

JDK-21

----------------------

Agent C

JDK-11
```

This can produce inconsistent build results.

Standardize versions wherever possible.

---

## Hardcoding JAVA_HOME

Avoid

```bash
export JAVA_HOME=/usr/lib/jvm/java-21
```

Instead,

use Jenkins Global Tool Configuration.

---

# Best Practices

- Configure JDK globally instead of hardcoding paths.
- Use meaningful names such as **JDK-17** and **JDK-21**.
- Standardize Java versions across build agents.
- Install JDK manually in production environments.
- Test applications before upgrading Java versions.
- Remove unused JDK versions periodically.

# Maven Configuration

## What is Maven Configuration?

**Maven Configuration** allows Jenkins administrators to configure one or more Apache Maven installations that can be used by Jenkins jobs and pipelines.

Apache Maven is one of the most popular build automation tools for Java applications.

It is used for:

- Dependency Management
- Compilation
- Unit Testing
- Packaging
- Plugin Management
- Artifact Creation

Instead of hardcoding the Maven installation path inside every Jenkinsfile, Jenkins allows administrators to configure Maven globally.

---

# Why Do We Need Maven Configuration?

Most Java applications use Maven to build the project.

A typical Java build includes:

```text
Download Dependencies

↓

Compile Source Code

↓

Run Unit Tests

↓

Generate JAR/WAR

↓

Publish Artifact
```

Without Maven, Jenkins cannot build Maven-based applications.

# Best Practices

- Configure Maven globally.
- Standardize Maven versions across all build agents.
- Pair Maven with the correct JDK version.
- Install Maven manually in production.
- Periodically update Maven to supported versions.
- Cache Maven dependencies to reduce build time.

---

# Troubleshooting

## Maven Command Not Found

Run

```bash
mvn -version
```

Install Maven if missing.

---

## Dependency Download Failed

Verify

- Internet connectivity
- Proxy configuration
- Repository availability
- Maven Central access

For enterprise environments, verify access to internal artifact repositories such as **JFrog Artifactory** or **Nexus Repository** if they are used.

---

## Wrong Maven Version

Run

```bash
mvn -version
```

Confirm Jenkins is using the configured Maven installation.

---

## Build Fails After Maven Upgrade

Check

- Plugin compatibility
- JDK compatibility
- Project `pom.xml`
- Maven plugin versions
