# Usage Statistics

## What are Usage Statistics?

**Usage Statistics** is a Jenkins setting that determines whether Jenkins sends anonymous usage information to the Jenkins project.

The collected information is **non-personal** and typically includes:

- Jenkins version
- Java version
- Operating System
- Installed plugins
- Platform information

This data helps the Jenkins community understand how Jenkins is being used and prioritize future improvements.

---

# Why Do We Need Usage Statistics?

The Jenkins community uses anonymous usage data to:

- Understand plugin popularity
- Improve compatibility
- Identify commonly used Java versions
- Support future feature development
- Plan LTS releases

For organizations, however, enabling this setting depends on internal security and compliance policies.

# Common Mistakes

## Assuming Jenkins Sends Source Code

Incorrect.

Jenkins does **not** upload:

- Git repositories
- Build artifacts
- Jenkinsfiles
- Credentials

---

## Enabling Without Approval

Some organizations require approval before allowing outbound communication.

---

Industries like banking or healthcare may prohibit telemetry from production systems.

---

# Best Practices

- Follow your organization's security policy.
- Disable Usage Statistics in highly regulated environments.
- Enable it only if your company permits anonymous telemetry.
- Regularly review outbound network access from the Jenkins controller.
- Document the decision as part of your Jenkins deployment standards.

# Markup Formatter

## What is Markup Formatter?

The **Markup Formatter** controls how text entered in Jenkins is displayed on web pages.

It determines whether Jenkins interprets text as:

- Plain text
- Safe HTML
- Markdown (through plugins)
- Other supported markup formats

This setting is mainly used for:

- Job descriptions
- Folder descriptions
- View descriptions
- Build descriptions
- Dashboard content

---

# Why Do We Need a Markup Formatter?

Without a markup formatter, Jenkins simply displays raw text.

Example

```text
Production Deployment

Please contact DevOps Team

support@company.com
```

It appears as simple text.

With a markup formatter, the same content can be formatted using headings, lists, bold text, links, and tables (depending on the formatter).

# Safe HTML

Safe HTML allows a limited set of HTML tags.

Example

```html
<b>Production Jenkins</b>

<ul>
<li>Java</li>
<li>NodeJS</li>
</ul>
```

Output

```text

• Java

• NodeJS
```

Unsafe HTML elements like JavaScript are automatically removed.

---

# Shell Configuration

## What is Shell Configuration?

The **Shell Configuration** in Jenkins specifies which shell interpreter should be used to execute shell commands on Linux or Unix-based Jenkins controllers and agents.

By default, Jenkins uses:

```text
/bin/sh
```

However, administrators can configure Jenkins to use a different shell such as:

- Bash
- Zsh
- Ksh
- Dash

This ensures all shell scripts execute consistently across all Jenkins jobs.

---

# Why Do We Need Shell Configuration?

Almost every CI/CD pipeline executes shell commands.

Examples include:

- Git clone
- Maven build
- Docker build
- Terraform apply
- Kubernetes deployment
- AWS CLI commands

Without a proper shell configuration, scripts may fail due to differences between shell interpreters.

---

# Common Mistakes

## Using Bash Syntax with /bin/sh

Example

```bash
arr=("one" "two")
```

This works in Bash but may fail in `/bin/sh`.

---

## Shell Not Installed

Configuration

```text
/bin/bash
```

If Bash is not installed,

every shell step fails.

---

## Different Shells on Different Agents

Example

```text
Agent A

/bin/bash

--------------------

Agent B

/bin/sh

--------------------

Agent C

/bin/zsh
```

This can produce inconsistent pipeline behavior.

---

# Best Practices

- Standardize on **/bin/bash** across Linux agents.
- Keep the same shell configuration on all build agents.
- Use portable scripts when possible.
- Validate shell scripts before production deployment.
- Avoid relying on shell-specific features unless required.
- Document shell requirements for shared libraries.

---

# Troubleshooting

## Script Works Locally but Fails in Jenkins

Check

```text
echo $SHELL
```

Verify the configured shell matches your local environment.

---

## Shell Not Found

Example

```text
/bin/bash

No such file or directory
```

Install Bash or update the shell path.

---

## Unexpected Script Behavior

Possible causes

- Wrong shell interpreter
- Different shell versions
- Shell-specific syntax

Verify the configured shell on every build agent.
