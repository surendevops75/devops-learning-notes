# Usage Statistics

## What are Usage Statistics?

**Usage Statistics** is a Jenkins setting that determines whether Jenkins sends anonymous usage information to the Jenkins project.

The collected information is **non-personal** and typically includes:

- Jenkins version
- Java version
- Operating System
- Installed plugins
- Platform information

This data helps the Jenkins community understand how Jenkins is being used and prioritize future improvements.

---

# Why Do We Need Usage Statistics?

The Jenkins community uses anonymous usage data to:

- Understand plugin popularity
- Improve compatibility
- Identify commonly used Java versions
- Support future feature development
- Plan LTS releases

For organizations, however, enabling this setting depends on internal security and compliance policies.

# Security Considerations

Although the data is anonymous,

many companies prefer not to allow any outbound communication from production servers.

Reasons include:

- Corporate security policy
- Regulatory compliance
- Network isolation
- Internal governance

---

# Common Mistakes

## Assuming Jenkins Sends Source Code

Incorrect.

Jenkins does **not** upload:

- Git repositories
- Build artifacts
- Jenkinsfiles
- Credentials

Only anonymous installation statistics are shared.

---

## Enabling Without Approval

Some organizations require approval before allowing outbound communication.

Always follow company security policies.

---

## Ignoring Compliance Requirements

Industries like banking or healthcare may prohibit telemetry from production systems.

---

# Best Practices

- Follow your organization's security policy.
- Disable Usage Statistics in highly regulated environments.
- Enable it only if your company permits anonymous telemetry.
- Regularly review outbound network access from the Jenkins controller.
- Document the decision as part of your Jenkins deployment standards.

---

# Troubleshooting

## Jenkins Cannot Send Statistics

Possible causes

- Firewall blocks internet access.
- Proxy configuration is missing.
- Internet connectivity is unavailable.

This does not affect pipeline execution.

```text
↓

↓

```

Save the configuration.

---

## No Internet Access

No action is required.

Jenkins continues functioning normally without sending statistics.

# Key Points

- Usage Statistics controls whether Jenkins sends anonymous installation information to the Jenkins project.
- No source code, credentials, build logs, or secrets are transmitted.

# Markup Formatter

## What is Markup Formatter?

The **Markup Formatter** controls how text entered in Jenkins is displayed on web pages.

It determines whether Jenkins interprets text as:

- Plain text
- Safe HTML
- Markdown (through plugins)
- Other supported markup formats

This setting is mainly used for:

- Job descriptions
- Folder descriptions
- View descriptions
- Build descriptions
- Dashboard content

---

# Why Do We Need a Markup Formatter?

Without a markup formatter, Jenkins simply displays raw text.

Example

```text
Production Deployment

Please contact DevOps Team

support@company.com
```

It appears as simple text.

With a markup formatter, the same content can be formatted using headings, lists, bold text, links, and tables (depending on the formatter).

# Plain Text

This is the default option.

Everything is displayed exactly as entered.

Example

```text
Welcome to Jenkins

Please follow CI/CD guidelines.
```

Displayed exactly the same.

Advantages

- Safe
- No HTML execution
- Recommended for secure environments

---

# Safe HTML

Safe HTML allows a limited set of HTML tags.

Example

```html
<b>Production Jenkins</b>

<ul>
<li>Java</li>
<li>NodeJS</li>
</ul>
```

Output

```text
Production Jenkins

• Java

• NodeJS
```

Unsafe HTML elements like JavaScript are automatically removed.

---

# Markdown

If the Markdown Formatter Plugin is installed,

you can write

```markdown
# Production Jenkins

## Build Rules

- Java
- Python
- NodeJS
```

Rendered as

```text
Production Jenkins

Build Rules

• Java

• Python

• NodeJS
```

# Security Considerations

Improper HTML rendering can introduce security risks.

Example

```html
<script>

alert("Hack")

</script>
```

A secure formatter removes dangerous scripts before displaying content.

This helps protect against:

- Cross-Site Scripting (XSS)
- Malicious HTML injection
- Unsafe JavaScript execution

# Common Mistakes

## Enabling Unsafe HTML

Allowing unrestricted HTML can expose Jenkins to security vulnerabilities.

Always use a safe formatter.

---

## Large Documentation

Avoid placing long documentation inside Jenkins.

Instead,

store detailed documentation in:

- GitHub
- GitLab
- Confluence
- Internal Wiki

Add only the documentation link in Jenkins.

---

## Using HTML for Everything

Markdown is usually easier to maintain than HTML.

Prefer Markdown where supported.

---

# Best Practices

- Use **Plain Text** when formatting is unnecessary.
- Use **Safe HTML** instead of unrestricted HTML.
- Prefer **Markdown** for readable job documentation.
- Keep descriptions short and relevant.
- Store detailed documentation in version control or a documentation platform.
- Never embed JavaScript or unsafe HTML.

---

# Troubleshooting

## HTML Not Rendering

Verify the selected formatter.

```text
Manage Jenkins

↓

System

↓

Markup Formatter
```

---

## Markdown Displays as Plain Text

Possible causes

- Markdown plugin is not installed.
- Plain Text formatter is selected.

Install the appropriate plugin and select it as the formatter.

---

## Formatting Looks Incorrect

Check for:

- Invalid Markdown syntax
- Unsupported HTML tags
- Plugin compatibility
