# Supply-chain Levels for Software Artifacts (SLSA)

## Introduction

Supply-chain Levels for Software Artifacts (SLSA, pronounced "salsa") is a security framework developed by Google and now maintained by the OpenSSF. It provides a set of standards and controls to improve the integrity of software throughout the software supply chain.

SLSA helps organizations prevent tampering, improve build integrity, and ensure that software artifacts are trustworthy.

---

## Why Do We Need SLSA?

Modern software is built using multiple tools, dependencies, CI/CD pipelines, and third-party components.

Without SLSA:

- Build systems can be compromised
- Malicious code may be injected
- Build artifacts cannot be trusted
- Provenance is unavailable
- Supply chain attacks become easier

SLSA provides measurable security levels to strengthen software supply chain security.

---

## What is SLSA?

SLSA is a security framework that defines requirements for securely building, signing, and distributing software.

Its goals are to:

- Protect source code
- Secure build systems
- Generate provenance
- Prevent artifact tampering
- Improve software integrity

SLSA is based on progressive maturity levels, allowing organizations to improve security over time.

# Core Components of SLSA

## Source

The source code must come from a trusted and version-controlled repository.

Examples

- GitHub
- GitLab
- Bitbucket

---

## Build System

The build process must be secure, isolated, and reproducible.

Examples

- GitHub Actions
- Jenkins
- GitLab CI
- Tekton

---

## Provenance

Provenance records how an artifact was built.

Typical information includes:

- Source Repository
- Commit ID
- Build Tool

- Dependencies

Requirements

- Isolated builds

↓

Requirements

- Two-person review

## Best Practices

- Use trusted source repositories
- Protect build systems
- Generate provenance for every build
- Digitally sign all production artifacts
- Verify signatures before deployment
- Generate SBOMs alongside provenance
- Implement reproducible builds where possible
- Enforce mandatory code reviews
- Store provenance securely
- Continuously audit build systems

---

## Common Mistakes

- Skipping provenance generation
- Using shared build servers without isolation
- Deploying unsigned artifacts
- Ignoring provenance verification
- Allowing direct manual deployments
- Not protecting CI/CD credentials
- Using outdated build tools
- Treating SLSA as a one-time implementation

# Production Interview Questions

## Question 1

### What is SLSA?

SLSA (Supply-chain Levels for Software Artifacts) is a security framework that defines standards for protecting software throughout the software supply chain.

---

## Question 2

### Why is SLSA important?

It improves build integrity, protects against supply chain attacks, and ensures software artifacts are trustworthy.

---

## Question 3

### What is build provenance?

Build provenance is metadata that records how, when, where, and by whom a software artifact was built.

---

## Question 4

### What are the four SLSA levels?

SLSA defines four maturity levels: Level 1, Level 2, Level 3, and Level 4, with increasing security requirements.

---

## Question 5

### What is the purpose of provenance?

Provenance provides traceability and allows organizations to verify the origin and integrity of software artifacts.

---

## Question 6

### Which tools support SLSA?

GitHub Actions, Tekton Chains, Sigstore Cosign, in-toto, Syft, and Trivy are commonly used to implement SLSA practices.

---

## Question 7

### How does SLSA improve DevSecOps?

It secures build pipelines, generates provenance, verifies software integrity, and strengthens software supply chain security.

---

## Question 8

### What is a reproducible build?

A reproducible build generates identical artifacts from the same source code, build environment, and configuration, improving trust and verification.

---

## Question 9

### What is the relationship between SLSA and SBOM?

An SBOM lists software components, while SLSA provides assurance about how the software was built and delivered. They complement each other.

---

## Question 10

### How would you implement SLSA in a CI/CD pipeline?

Protect the source repository, use a trusted build system, generate SBOM and provenance, sign artifacts, verify signatures and provenance before deployment, and continuously audit the pipeline.

---

# Key Takeaways

- SLSA is a framework for securing the software supply chain.
- It focuses on build integrity, provenance, and artifact authenticity.
- Provenance records how software was built.
- SLSA defines four maturity levels for progressively stronger security.
- Integrating SLSA into DevSecOps pipelines significantly reduces the risk of software supply chain attacks.
